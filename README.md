# ESP32-S3 FOTA LED Study Starter

ESP32-S3에서 **내장 LED를 주기적으로 깜빡이고**,  
**Wi-Fi에 연결한 뒤**,  
**OTA(Over-The-Air) 방식으로 펌웨어를 업데이트할 수 있도록 만든 학습용 프로젝트**다.

처음에는 LED를 **1초마다 깜빡이게** 실행하고,  
이후 OTA로 새 펌웨어를 올려 **2초마다 깜빡이게** 바꾸는 식으로 FOTA 흐름을 익히는 것이 목표다.

---

## 프로젝트 목적

이 프로젝트는 단순히 LED를 깜빡이는 예제가 아니라,  
작은 임베디드 프로그램을 **기능별로 나누고**,  
**부팅 → LED 동작 → Wi-Fi 연결 → OTA 대기** 흐름을 이해하기 위한 구조로 설계했다.

특히 아래 내용을 배우기 좋다.

- 설정값과 로직 분리
- LED 제어 모듈화
- Wi-Fi 연결 및 재연결 구조
- OTA 준비 및 업데이트 처리 흐름
- `delay()`를 줄이고 `millis()` 기반으로 동작시키는 이유


### 파일 역할

| 파일명 | 역할 |
| --- | --- |
| `main.ino` | 전체 앱의 시작점. 각 모듈을 호출하는 메인 제어 파일 |
| `app_config.h` | 프로젝트 설정값 모음. LED, Wi-Fi, OTA 관련 값 정의 |
| `led_blinker.*` | LED 초기화, ON/OFF, 주기적 토글 담당 |
| `wifi_manager.*` | Wi-Fi 연결, 상태 확인, 재연결 담당 |
| `ota_manager.*` | OTA 준비, OTA 요청 처리, 진행 로그 담당 |

## 전체 동작 흐름

ESP32-S3가 부팅되면 아래 순서로 동작한다.

1. 시리얼 포트를 연다.
2. 현재 프로젝트 설정 정보를 출력한다.
3. LED 모듈을 초기화한다.
4. LED 깜빡임 주기를 설정한다.
5. Wi-Fi 연결을 시작한다.
6. OTA 기능을 준비한다.
7. 이후 반복 루프에서
    - LED 깜빡임 갱신
    - Wi-Fi 상태 확인
    - 필요 시 OTA 재준비
    - OTA 요청 감시
        
        를 계속 수행한다.

## 모듈별 설명

---

### `main.ino`

`main.ino`는 프로젝트 전체를 지휘하는 파일이다.

실제 LED 제어나 Wi-Fi 내부 구현을 직접 하지 않고,

각 모듈을 **언제 호출할지** 결정한다.

### 주요 역할

- 시리얼 시작
- 설정값 출력
- LED 초기화
- Wi-Fi 시작
- OTA 시작
- 반복 루프에서 각 모듈 update 호출

### 핵심 포인트

이 파일은 **“반장” 역할**만 한다.

세부 구현을 각 모듈로 나눠서,

전체 흐름이 한눈에 보이도록 만든 구조다.

즉, `main.ino`는 다음만 책임진다.

- **무엇을**
- **언제**
- **어떤 순서로**

실행할지 정하는 것

---

### `app_config.h`

`app_config.h`는 프로젝트에서 바뀔 수 있는 설정값을 모아둔 파일이다.

예를 들어 다음 값들이 들어간다.

- 프로젝트 이름
- 시리얼 속도
- Wi-Fi SSID / 비밀번호
- OTA 호스트 이름
- OTA 비밀번호 사용 여부
- LED 핀 번호
- LED ON 레벨
- LED 깜빡임 주기

### 왜 따로 분리했는가

설정값을 코드 본문과 분리하면 아래 장점이 있다.

- 핀 번호 변경이 쉬움
- Wi-Fi 정보 수정이 쉬움
- OTA 이름 변경이 쉬움
- LED 주기 변경이 쉬움
- 다른 프로젝트로 재사용하기 쉬움

즉,

**“동작 로직”과 “환경 설정”을 분리하기 위해서**다.

---

### `led_blinker`

`led_blinker` 모듈은 LED 제어만 담당한다.

### 주요 역할

- LED 핀 초기화
- LED ON / OFF 처리
- 깜빡임 주기 설정
- 일정 시간마다 LED 상태 토글

### 내부 동작 방식

이 모듈은 `delay()` 방식이 아니라

`millis()`를 이용한 **비차단 방식**으로 LED를 깜빡인다.

즉,

매 loop마다 현재 시간을 확인하고,

설정한 주기가 지났을 때만 LED를 ON/OFF 전환한다.

### 왜 이렇게 했는가

`delay(1000)` 같은 방식은 이해는 쉽지만,

그 시간 동안 CPU가 멈춘다.

그러면 아래 작업이 같이 멈출 수 있다.

- Wi-Fi 상태 확인
- OTA 요청 감시
- 이후 추가될 다른 작업

그래서 이 프로젝트는 LED를 깜빡이면서도

다른 기능이 동시에 동작할 수 있도록

`millis()` 기반 구조를 사용했다.

### 추가 고려 사항

보드에 따라 내장 LED가 일반 디지털 LED가 아닐 수도 있다.

그래서 설정값으로 단순 LED 여부와 RGB 여부를 구분해,

현재 방식으로 제어 가능한 경우에만 동작하도록 만들었다.

---

### `wifi_manager`

`wifi_manager` 모듈은 Wi-Fi 연결을 담당한다.

### 주요 역할

- Station 모드 설정
- 공유기에 연결 시도
- 연결 상태 출력
- 연결이 끊기면 재연결 시도
- 현재 연결 여부 반환

### 내부 동작 방식

부팅 직후 Wi-Fi 연결을 시도하고,

반복 루프에서는 현재 연결 상태를 계속 확인한다.

만약 연결이 끊기면 바로 무한 재시도하지 않고,

설정한 간격 이후 다시 연결을 시도한다.

### 왜 이렇게 했는가

Wi-Fi는 실제 환경에서 자주 끊길 수 있다.

그래서 한 번만 연결하고 끝내는 방식보다,

**끊기면 다시 붙는 구조**가 더 실용적이다.

이 구조가 필요한 이유는 OTA 때문이다.

OTA는 네트워크가 살아 있어야 동작한다.

즉, Wi-Fi 관리가 안정적이어야 OTA도 안정적으로 사용할 수 있다.

---

### `ota_manager`

`ota_manager` 모듈은 OTA 기능만 담당한다.

### 주요 역할

- Wi-Fi 연결 여부 확인
- OTA 호스트 이름 설정
- 필요 시 OTA 비밀번호 설정
- OTA 시작 / 진행 / 종료 / 에러 로그 처리
- 반복 루프에서 OTA 요청 감시

### 내부 동작 방식

Wi-Fi가 연결되지 않은 상태에서는 OTA를 시작하지 않는다.

Wi-Fi가 연결된 뒤에만 `ArduinoOTA.begin()`을 호출해 OTA를 준비한다.

반복 루프에서는 `ArduinoOTA.handle()`를 계속 호출해

업데이트 요청이 들어오는지 감시한다.

### 왜 이렇게 했는가

OTA는 네트워크를 통해 펌웨어를 받는 기능이므로,

Wi-Fi가 연결되지 않으면 의미가 없다.

따라서 OTA 모듈은

**“Wi-Fi 연결 완료 후 활성화”**

라는 순서를 따르도록 설계했다.

또한 업데이트 과정에서 아래 상태를 로그로 구분한다.

- 시작
- 진행률
- 종료
- 에러

이렇게 하면 디버깅이 쉬워진다.

## 실행 순서 요약

### `setup()`

전원이 켜졌을 때 한 번만 실행된다.

- 시리얼 시작
- 프로젝트 정보 출력
- LED 초기화
- LED 주기 설정
- Wi-Fi 시작
- OTA 준비 시도

### `loop()`

그 이후 계속 반복된다.

- LED 깜빡일 시간인지 확인
- Wi-Fi 상태 점검
- Wi-Fi가 연결됐는데 OTA가 아직 준비 안 된 경우 OTA 재시작
- OTA 요청 감시

## 왜 이런 구조로 나눴는가

이 프로젝트는 학습용이지만,

나중에 다른 프로젝트에도 그대로 확장할 수 있도록

구조를 분리했다.

### 장점

### 1. 읽기 쉽다

각 파일이 자기 역할만 담당하므로,

초보자도 어디를 봐야 할지 명확하다.

### 2. 수정이 쉽다

LED만 바꾸고 싶으면 `led_blinker`만 보면 된다.

Wi-Fi 문제를 보고 싶으면 `wifi_manager`만 보면 된다.

### 3. 재사용이 쉽다

나중에 LED 대신 릴레이, 센서, 모터를 붙여도

전체 구조는 그대로 유지할 수 있다.

### 4. 유지보수가 쉽다

기능이 섞여 있지 않아서 문제를 찾기 쉽다.

즉,

**작게 나누고, 각자 자기 일만 하게 만든 구조**다.
